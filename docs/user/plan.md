# 视频系列规划：Modding Toolkit 人物外观 MOD 制作全流程

**风格**: 实操演示为主（屏幕录制 + 旁白）  
**每集时长**: 10-15 分钟  
**平台**: Blender 4.x（推荐 4.3.2）  
**讲解语言**: 中文

---

## 系列总览

七个视频覆盖从导入模型到最终导出发布的完整管线。姿态修正在最前面，骨架对齐和顶点组转换紧随其后，物理处理、材质贴图、批量导出各自成集，最终以进阶技巧收尾。

```
视频1        →   视频2       →   视频3     →   视频4
姿态修正与模型准备   骨架选择与对齐    顶点组转换      物理骨骼处理
（先对姿态）       （再对上骨骼）    （再通顶点组）   （再处理物理）

视频5        →   视频6       →   视频7
材质与贴图处理      批量导出与发布    进阶技巧与自定义预设
```

- 视频 1-4 是**主线流程**，按顺序必须依次完成
- 视频 5-6 是**并行/后置**环节，可单独观看
- 视频 7 是**锦上添花**，提升效率和解决特殊需求

## 视频 1：姿态修正与模型准备（最先做）

**目标**：将导入的任意来源模型调整到与目标游戏骨架大致匹配的姿态和体型，为后续骨架对齐打好基础。

### 内容纲要

| 序号 | 环节 | 涉及功能 | 时间 |
|------|------|----------|------|
| 1.1 | 模型导入与检查 | Blender 通用操作（导入 FBX/PMX、检查骨架层级、观察参考模型姿态） | 2min |
| 1.2 | 选择正确的姿态工具 | 三种姿态修正工具的适用场景对比 | 2min |
| 1.3 | 方向计算 | 插件「方向计算」：仅旋转上臂到水平方向（A-Pose→T-Pose 的简易方案） | 2min |
| 1.4 | RE Engine 矩阵归零 | 插件「RE Engine 矩阵归零」：重置肢体骨骼旋转矩阵（适用 RE 引擎游戏，生化9除外） | 2min |
| 1.5 | 姿态变换记录器 | 插件「录制变换」+「正向/逆向」：录制同类型骨架的 A→B 姿态差，一劳永逸复用 | 3min |
| 1.6 | 手动补充调整 | Blender 通用操作：根骨骼的缩放/旋转/位移手工矫正 | 1min |

### 关键讲解点

- **为什么姿态要先修正**：来源模型和目标游戏姿态差异越大，后续「对齐骨骼」的偏差越大。先对姿态，再对骨骼。
- **方向计算 vs 矩阵归零 vs 录制器**的选择逻辑：
  - MMD 等简单 A-Pose 模型 → 先用「方向计算」旋转上臂，配合手动微调
  - 制作 RE 引擎游戏 MOD → 用「RE Engine 矩阵归零」，一步到位重置肢体
  - 复杂/特殊姿态且有多份同类模型 → 用「姿态变换记录器」录一次，处处复用
- **姿态变换记录器**原理：计算每根骨骼相对于父级的局部旋转差（delta），忽略无变化骨骼。保存为 JSON，插件更新不丢失。

### 易错点

| 问题 | 现象 | 解决 |
|------|------|------|
| 方向计算只旋转上臂 | 下肢仍保持 A-Pose | 这是预期行为，下肢需后续手动调整或配合录制器 |
| RE9 用矩阵归零 | 骨骼方向异常 | RE9 骨骼矩阵体系不同，此功能已标注「生化9除外」 |
| 录制时骨骼名不匹配 | 找不到对应骨骼 | 在姿态转换区选择正确的「骨架预设」作为桥接 |
| 录制时只有一个骨架 | 无法录制 | 需要同时打开两个同类型但不同姿态的骨架 |

### 涉及的面板区域

- 姿态转换 (Pose Convert) 区块（全部功能）

## 视频 2：骨架选择与对齐

**目标**：理解 X/Y 预设架构，将目标游戏骨架对齐到来源模型的骨架上。

### 内容纲要

| 序号 | 环节 | 涉及功能 | 时间 |
|------|------|----------|------|
| 2.1 | X/Y 预设概念 | 图解：X=来源模型类型，Y=目标游戏；预设下拉菜单的分类浏览与切换 | 2min |
| 2.2 | 选择 X 和 Y 预设 | 面板「来源预设 (X)」和「目标游戏 (Y)」下拉框的操作 | 1min |
| 2.3 | 导入目标游戏骨架 | 如何获取或导入目标游戏的标准骨骼（MHWs/MHWI/RE4/RE9） | 1min |
| 2.4 | 对齐骨骼（核心） | 插件「对齐骨骼 [X+Y, 双骨架]」：先选源骨架→Ctrl 加选目标骨架，点击对齐 | 3min |
| 2.5 | 通用对齐工具（三种模式） | 基础工具区「位置 / 位置+扭转 / 完全」：不依赖 X+Y 预设的跨骨架对齐 | 2min |
| 2.6 | 映射详情预览 | 展开面板查看每根标准骨骼的匹配状态（✓ 匹配成功 / ✗ 缺失） | 1min |

### 关键讲解点

- **X/Y 预设的模糊匹配**：骨骼名中的 `_`、`.`、空格在匹配时被归一化。`UpperArm.L` 和 `UpperArm_L` 视为等价。
- **对齐骨骼的内部逻辑**（刚性对齐）：目标骨骼头部移动到来源骨头部位置，尾部同步偏移（保持骨骼方向不变），然后递归传递偏移给所有子孙骨骼。父级先移，子级后调。
- **通用对齐 vs X+Y 对齐**：前者按同名骨骼直接匹配，不需要预设；后者通过 X→标准键→Y 桥接，支持不同命名体系。
- **58 个标准骨骼名**覆盖范围：骨盆→脊椎→颈部→头部，锁骨→上臂→前臂→手掌，大腿→小腿→脚→脚趾，五根手指各三节（thumb/index/middle/ring/pinky）。spine_03 为可选骨骼。
- **skip_snap**：某些 Y 预设中特定骨骼标记了 `skip_snap: true`（如某些游戏的根骨骼），对齐时自动跳过。

### 易错点

| 问题 | 现象 | 解决 |
|------|------|------|
| 选中顺序错误 | 对齐方向反了 | 先选来源骨架，再 Ctrl 加选目标骨架。目标骨架必须是黄色激活状态 |
| spine_03 无法对齐 | 预览显示 spine_03 缺失 | spine_03 是可选骨，某些游戏（MHWI/MHWs）不支持，会自动降级到 spine_02 |
| 对齐后姿态仍然不对 | 网格变形异常 | 回到视频1检查姿态修正；通用对齐模式「仅位置」不会改变骨骼方向 |
| 骨骼数量不对应 | 大量 ✗ | 检查来源模型是否与 X 预设匹配（VRChat 标准模型用 VRChat 预设等） |

### 涉及的面板区域

- 通用标准转换区块（来源/目标预设、对齐骨骼）
- 基础工具区（位置/位置+扭转/完全 对齐按钮）
- 映射详情预览（展开折叠区）

## 视频 3：顶点组转换与基础清理

**目标**：一键将来源模型网格的顶点组重命名为目标游戏格式，完成蒙皮权重的迁移。

### 内容纲要

| 序号 | 环节 | 涉及功能 | 时间 |
|------|------|----------|------|
| 3.1 | 重命名顶点组（核心） | 插件「重命名顶点组 [X+Y]」：选中全部网格→一键转换所有顶点组名 | 3min |
| 3.2 | 辅助骨骼的自动合并 | 预设中 aux 列表的骨骼权重如何被自动并入主骨骼 | 1min |
| 3.3 | 骨骼显示控制 | 三态切换「全显 / 仅基础骨 / 仅物理骨」：方便检查转换结果 | 1min |
| 3.4 | 基础骨骼改名 | 骨架编辑模式下将基础骨名从来源名改为目标游戏名 | 1min |
| 3.5 | 通用清理工具 | 基础工具区：扭转归零 / 骨链简化 / 合并到激活骨 / 合并链到激活链 | 3min |
| 3.6 | 映射详情复查 | 再次展开预览，确认所有基础骨骼均显示 ✓ | 1min |

### 关键讲解点

- **顶点组重命名的内部流程**：
  1. 对于每个标准骨骼键（如 `upperarm_L`），从 X 预设获取来源主骨名+辅助骨名列表
  2. 从 Y 预设获取目标主骨名
  3. 在网格上找到实际存在的顶点组（模糊匹配）
  4. 将辅助骨骼的权重合并到主骨骼，然后将主骨骼重命名为目标名
  5. 如果主骨不在网格上但辅助骨在，辅助骨直接合并到目标名
  6. spine_03 的特殊处理：目标游戏不支持时自动降级到 spine_02
- **骨链简化**原理：选中链上多个骨骼，按「两两配对」合并（保留前一，后一的权重并入前一），自动跳过无顶点权重的尾骨。例如 5 根骨骼 → 合并为 3 根。
- **合并到激活骨 vs 合并链到激活链**：前者合并任意选中的骨骼；后者识别多条链的链首，按位置一一对应合并，多余骨骼归入链末。

### 易错点

| 问题 | 现象 | 解决 |
|------|------|------|
| 选了骨架而非网格 | 顶点组没有变化 | 必须选中网格对象（Mesh），而非骨架（Armature） |
| 改名时目标名被占用 | 某些骨骼名带 `_old` 后缀 | 系统自动给冲突骨骼加 `_old` 后缀避让，检查并手动清理 |
| 骨链简化结果不对 | 骨骼配对错误 | 骨骼必须属于同一条父链，系统按编辑模式选中顺序自动分组 |
| 合并到激活骨搞反 | 激活骨骼被删除 | 最后点击的那根骨骼是激活骨（保留目标），不要再点别处 |

### 涉及的面板区域

- 通用标准转换区块（重命名顶点组）
- 基础工具区（全部 8 项功能）
- 通用标准转换 → 映射详情预览

## 视频 4：物理骨骼处理

**目标**：解决物理骨骼（头发、裙子、尾巴等）的保留或清理问题。提供两条路线。

### 内容纲要

| 序号 | 环节 | 涉及功能 | 时间 |
|------|------|----------|------|
| 4.1 | 两条路线概览 | 路线 A（不保留，简单）vs 路线 B（保留，进阶），根据需求选择 | 1min |
| 4.2 | **路线 A** - 物理权重降级 | 插件「物理权重降级 [X]」：物理骨权重沿父链向上合并到最近身体骨 | 2min |
| 4.3 | **路线 A** - 剔除非基础骨骼 | 插件「剔除非基础骨骼 [X]」：删除所有不在预设中的骨骼（即删除物理骨） | 1min |
| 4.4 | **路线 B** - 移植物理骨骼 | 插件「移植物理骨骼 [X+Y, 双骨架]」：完整复制物理骨+自动生成 _End 末端骨+竖直化重置+智能重建父子关系 | 3min |
| 4.5 | **路线 B** - 物理链可视化标记 | 插件「刷新骨骼颜色」：四色标记（链首=天蓝，支链头=紫色，主链延伸=琥珀色，普通=深蓝） | 1min |
| 4.6 | **路线 B** - 手动调整链结构 | 插件「标记为主链延伸」+「清除标记」：在分叉处指定主链方向 | 1min |
| 4.7 | **路线 B** - 清理功能根骨 | 插件「合并到父骨」：将 hair_root 等功能性连接器骨骼的权重归入父骨 | 1min |
| 4.8 | MHWI 特殊处理 | 「拆分物理骨」+「一键重命名」：解决 255 骨骼上限，按部位拆分+重命名为 MhBone_xxx 格式 | 2min |

### 关键讲解点

- **何时选路线 A**：目标游戏不需要物理模拟、或者物理骨骼过于复杂无法移植、或者只是简单的外观替换。
- **何时选路线 B**：需要保留物理效果（裙子飘动、头发摆动等），且来源模型的物理骨结构相对完整。
- **移植物理骨骼的内部流程**：
  1. 筛选：源骨架中不在 X 预设内的骨骼全部视为物理骨
  2. 复制：每根物理骨的世界坐标直接复制到目标骨架
  3. 自动生成 `_End` 末端骨：检测叶骨（无物理子骨）和分叉无主链标记处，在源骨骼尾部位置生成
  4. 竖直化：所有移植骨+End 骨重置为竖直向上（Z+），断开 use_connect
  5. 智能重建父级：物理骨父子→物理骨父子；物理骨父级为映射骨→通过标准键桥接到目标骨架对应骨；找不到直接映射→沿祖先链向上查找
  6. 复制 chain_role 属性：从源骨骼复制到目标骨骼
- **物理权色标记系统**：
  - chain_role = "head"：天蓝色，链首骨骼（父骨不在物理骨集合中）
  - chain_role = "branch_head"：紫色，分叉支链的起点（父骨是分叉骨骼且未标记为主链）
  - chain_role = "main_continue"：琥珀色，分叉处被手动标记为主链延伸的子骨
  - 无标记 / body / _End：深蓝色，普通物理体骨和末端骨
- **MHWI 拆分逻辑**：按解剖区域（头/臂/躯干/腿）将物理骨归类，超过 body 容量（255-基础骨骼数）时自动拆分为多个子骨架（_arm/_wst/_leg），每个子骨架独立命名。

### 易错点

| 问题 | 现象 | 解决 |
|------|------|------|
| 降级后权重到错误骨骼 | 物理骨权重跑到不相关的身体骨 | 物理权重沿父链向上查找，确保父链在降级前完整；骨骼层级如果混乱需先修复 |
| 移植后 _End 骨位置不对 | End 骨飞在很远的位置 | End 骨的 head = 源骨骼的 tail（世界坐标），如果源骨长度异常则 _End 位置也会异常 |
| 移植需要双预设都加载 | 按钮不可用或报错 | 必须在通用转换区选择有效的 X 和 Y 预设 |
| MHWI 未拆分就重命名 | 骨架超 255 根 | 先运行「拆分物理骨」→ 再运行「一键重命名」 |
| 分叉处自动生成不需要的 _End | 多了不必要的 End 骨 | 在分叉的子骨上手动「标记为主链延伸」，系统就不会在此位置生成 _End |

### 涉及的面板区域

- 通用标准转换 → 实验性功能（全部功能）
- MHWI Tools（对齐非物理骨骼、拆分物理骨、一键重命名）

## 视频 5：材质与贴图处理

**目标**：配置和生成游戏格式的材质文件（MDF2/MRL3）和贴图文件（.tex），使 MOD 在游戏中正常渲染。

### 内容纲要

| 序号 | 环节 | 涉及功能 | 时间 |
|------|------|----------|------|
| 5.1 | MDF2 概念简介 | MDF2 集合、贴图槽位、Natives Root 路径 | 2min |
| 5.2 | MDF2 处理器 - PBR 输入配置 | 7 种 PBR 输入逐个说明（基础色/Alpha/法线/粗糙度/金属度/AO/自发光） | 3min |
| 5.3 | MDF2 处理器 - 通道选择与反相 | Alpha/粗糙度/金属度/AO 可指定来源通道（R/G/B/A）+ 反相开关 | 2min |
| 5.4 | MDF2 处理器 - 四种槽位模式 | PBR转换(COMPOSE) / 直接选择(DIRECT) / 默认空贴图(DEFAULT) / 不修改(SKIP) | 2min |
| 5.5 | MDF2 处理器 - 高级功能 | GL→DX 法线翻转、刷新自动检测空贴图、复制粘贴材质配置、状态持久化 | 2min |
| 5.6 | MDF2 生成器 | 从 Blender Principled BSDF 节点自动生成 MDF2 集合 + 贴图 | 1min |
| 5.7 | MRL3 处理/生成器（MHWI） | MHWI 专用 MRL3 材质处理 | 1min |

### 关键讲解点

- **7 种 PBR 输入及其颜色空间**：
  - 基础色、自发光 → sRGB（输出 BC7_UNORM_SRGB）
  - 法线、粗糙度、金属度、AO → Linear（输出 BC7_UNORM）
  - Alpha → 取决于槽位
- **通道选择与反相的典型用例**：
  - 贴图的 R 通道存的是「平滑度」→ 选择 R 通道 + 开启反相 = 得到「粗糙度」
  - 贴图的 A 通道存的是「绝缘度」→ 选择 A 通道 + 开启反相 = 得到「金属度」
  - 金属度贴图的 A 通道存的是 AO → 直接选择 A 通道
- **四种槽位模式的选择逻辑**：
  - **PBR转换**（最常用）：材质有对应贴图且需要合成通道 → 自动按游戏要求的 R/G/B/A 布局生成
  - **直接选择**：已有现成的 DDS/TEX 文件，或需要单通道贴图直接使用
  - **默认空贴图**：此槽位不需要自定义贴图，写入游戏内置空贴图路径
  - **不修改**：保持材质中原有的路径不变（比如复用原游戏纹理）
- **GL→DX 法线翻转**：GL 格式法线的 G（绿色）通道方向与 DX 格式相反。勾选后在处理时自动翻转 G 通道。
- **复制粘贴材质配置**：对一个材质配置完 PBR 输入后，可以复制配置并粘贴到其他材质，避免重复操作。
- **状态持久化**：切换到不同的 MDF2 集合时，每个集合的配置独立保留，不会丢失。

### 易错点

| 问题 | 现象 | 解决 |
|------|------|------|
| Natives Root 路径设错 | 所有输出的路径都不对 | 设置为游戏 Natives 目录的上级（即含有 `natives\STM\` 路径的那一级） |
| 粗糙度忘记反相 | 材质光泽异常（过亮或过暗） | 如果输入是平滑度/光泽度贴图，必须开启 R 通道反相 |
| 金属度忘记反相 | 金属区域反了 | 如果输入是绝缘度（dielectric）贴图，必须开启 A 通道反相 |
| 法线翻转忘记勾选 | 光照方向异常 | 检查来源贴图是 GL 还是 DX 格式（GL 法线 G 通道朝上，DX 法线 G 通道朝下） |
| 刷新未触发空贴图检测 | 默认空贴图模式没有自动设置 | 切换到 MDF2 集合后需要点击「刷新」按钮触发自动检测 |
| 没有 texconv | 转换失败 | RE Mesh Editor 应自带 texconv，确认 RE Mesh Editor 已正确安装 |

### 涉及的面板区域

- MHWilds / RE4 / RE9 Tools 中的「MDF2 处理器」和「MDF2 生成器」按钮（弹出独立对话框）
- MHWI Tools 中的「MRL3 处理器」和「MRL3 生成器」按钮

## 视频 6：批量导出与发布

**目标**：完成最终的批量导出，生成游戏可直接使用的完整 MOD 文件包。

### 内容纲要

| 序号 | 环节 | 涉及功能 | 时间 |
|------|------|----------|------|
| 6.1 | 集合绑定 | 在 Blender 中为每个部位+文件类型绑定对应的集合 | 2min |
| 6.2 | MHWs 批量导出 | 选择装备包→套装种类→装备→打开导出对话框→导出 mesh/MDF2/chain2/clsp | 3min |
| 6.3 | BoneSystem 导出（MHWs） | 配置 FBXSkel 定义名+骨架+隐藏/绑定选项，生成 .fbxskel.7 和 BoneSystem JSON | 2min |
| 6.4 | MHWI 批量导出 | 选择装备包→位阶（HR/MR/SP）→性别→装备→导出 mod3/mrl3/ctc/ccl | 2min |
| 6.5 | RE4/RE9 批量导出 | 选择导出方案 JSON、简化模式、假头法（RE4）、空模型选项 | 2min |
| 6.6 | 一键创建 RE Chain | 检测链首骨骼→选择 Chain Collection→创建 Chain Settings + Chain Group（线性/分叉/各自独立/共享模式） | 2min |
| 6.7 | 导出后的文件检查 | 目录结构验证、路径确认、常见遗漏检查 | 1min |

### 关键讲解点

- **MHWs 多变体共享绑定**：4 种变体（男猎男套 mm / 男猎女套 mf / 女猎男套 fm / 女猎女套 ff）共享同一套集合绑定。配置一次，所有变体通用。依赖 JSON 中的 `parts_mask` 字段跳过不存在的部位。
- **MHWI 位阶选择**：HR=下位+上位装备（共用一套模型）、MR=大师位装备（冰原）、SP=整套幻化（独立幻化套装，可能包含头部/头发模型）。
- **BoneSystem 导出**需要设置：
  - `FBXSkel 定义名`：写入 `FbxPath` 字段 + `.fbxskel.7` 文件名（如 `ch03_000_9000`）
  - `骨架`：指向场景中的 MHWs 角色骨架对象
  - `隐藏面部/头发/投射器`：导出时选择性忽略
  - `绑定面部`：是否在 BoneSystem JSON 中绑定面部数据
  - `绑定部位`：头盔(=1) 或 身体(=2)
- **RE9 简化模式**：不同于逐条目绑定，简化模式下只需按组绑定一个集合，"empty" 类型条目自动使用内置空文件。
- **空模型选项**：未绑定集合的槽位在导出时自动复制内置空白文件（blank mesh/MDF2/chain2/clsp），避免导出中断。
- **RE Chain 创建的两种模式**：
  - **各自独立**（SEPARATE）：每条物理链使用自己独立的 Chain Settings
  - **共享同一**（SHARED）：所有链共用同一个 Chain Settings
- **MHWI 的 CTC Chain vs MHWs/RE4/RE9 的 RE Chain**：不同的链系统，需要不同的依赖插件（MHW Model Editor vs RE Chain Editor），操作流程类似。
- **MHWI CTC Chain 幂等性**：已创建的链不会被重复创建（通过名称匹配已有链头骨骼名），存在分叉的链会被跳过并报告。

### 易错点

| 问题 | 现象 | 解决 |
|------|------|------|
| 依赖插件未安装 | 导出按钮灰色不可用 | 安装 RE Mesh Editor（MHWs/RE4/RE9）或 MHW Model Editor（MHWI） |
| 集合绑定错位 | 导出文件内容和预期不符 | 逐个检查导出对话框中的绑定表，确保每个槽位对应正确的集合 |
| parts_mask 未配置 | JSON 中不存在的部位被要求绑定 | 在装备包 JSON 中为非必选部位添加 `parts_mask` 条目 |
| BoneSystem 导出失败 | fbxskel 生成报错 | 确认已选择正确的骨架对象（ARMATURE 类型），且骨架中包含与名称匹配的骨骼 |
| RE Chain 集合中无 Header | 创建链时报错 | 先用 RE Chain Editor 在集合中手动创建一个 Chain Header |
| 空模型选项状态混乱 | 不需要的栏位也被填充了空文件 | 按需开关，RE9 默认开启、MHWs 默认关闭 |

### 涉及的面板区域

- MHWI Tools（批量导出、批量导入）
- MHWilds Tools（MHWs Batch Exporter、一键创建 RE Chain）
- RE4 Tools（RE4 Batch Exporter、假头法、一键创建 RE Chain）
- RE9 Tools（RE9 Batch Exporter、一键创建 RE Chain）

## 视频 7：进阶技巧与自定义预设

**目标**：掌握插件的高级功能，包括自定义骨骼预设、特殊游戏工具和效率提升技巧。

### 内容纲要

| 序号 | 环节 | 涉及功能 | 时间 |
|------|------|----------|------|
| 7.1 | 可视化预设编辑器 | 从零创建自定义 X/Y 预设：清空初始化→吸管选取骨骼→L→R 自动镜像→批量添加辅助骨→保存 | 3min |
| 7.2 | 预设管理与发布 | 转换预设（X↔Y 互复制）、打开预设文件夹、管理 JSON 文件 | 1min |
| 7.3 | 姿态变换记录器深度使用 | 录一次到处用：不同骨架名通过预设桥接的详细原理 | 1min |
| 7.4 | RE4 假头法 (FakeBone) | 一键生成 RE4 角色身体+手指 End 骨骼：内部两段约束法流程 | 2min |
| 7.5 | MHWs 面部处理 | Endfield 面部改名（118条目映射）+ 面部权重简化（按比例合并 60% 权重的逻辑） | 2min |
| 7.6 | RE9 同步子级朝向 | 选中一批链首骨骼，递归将其所有子骨骼的朝向和 Roll 同步到父骨骼 | 1min |
| 7.7 | 通用对齐 - 不依赖预设 | 两个任意骨架按名称完全对齐，不依赖 X+Y 预设 | 1min |
| 7.8 | 补充基础工具 | 镜像对齐 X（左右对称）、添加尾骨 | 1min |
| 7.9 | 预设文件格式详解 | JSON 结构说明：main/aux/exclude 字段的含义和最佳实践 | 1min |

### 关键讲解点

- **预设编辑器**的操作流程：
  1. 选择编辑模式（X=来源 / Y=目标）
  2. 点击「清空并初始化列表」→ 58 个标准骨骼槽位全部生成
  3. 在 3D 视口中选中骨架，点击每个槽位的吸管图标（👁）选取对应骨骼
  4. 需要辅助骨骼的工具骨骼，点击 `+` 按钮批量添加
  5. 填完左侧后点击「L→R」一键镜像到右侧
  6. 输入保存名称，点击「保存」
  7. 可通过「复制为 Y 预设」或「复制为 X 预设」进行 X↔Y 跨目录转换
- **假头法**的内部逻辑：
  1. 从原生 fbxskel 文件导入游戏原生骨架作为参考骨架（ruler）
  2. **身体阶段**：对 ruler 应用旋转约束→可视化变换应用→创建 End 骨→应用缩放+位置约束→只保留 End 骨→合并回用户骨架→重建父级
  3. **手指阶段**：类似流程，但针对手指骨骼（Index/Thumb/Middle/Ring/Pinky），End 骨命名规则更复杂（如 `L_Palm_endP`, `L_IndexF1_endI`）
- **面部权重简化**的核心逻辑：
  - 使用 `_transfer_partial` 函数，将源顶点组的权重按比例（60%）分配给目标，剩余 40% 保留在源
  - 例如：细分的 L_LoEyeLid_B_LOD00, L_LoEyeLid_A_LOD00 → 60% 给 L_LoEyeLid_LOD01
  - 权重上限 clamp 到 1.0，防止超标
  - 嘴唇的权重拆分更精细（有的同时分配到两个目标，各 30%）
- **RE9 同步子级朝向**：遍历选中骨骼的递归所有子孙，将 tail 方向对齐到 parent 方向、roll 同步。不能同时选中父子骨骼。
- **JSON 预设中的字段**：
  - `main`：主骨骼候选列表，按优先级排列。系统遍历列表在骨架上查找，第一个匹配成功即使用
  - `aux`：辅助骨骼列表，其权重在转换时自动合并到主骨骼
  - `exclude`：排除列表中的骨骼不会被当作物理骨处理（如武器根骨、相机根骨等功能性骨骼）

### 易错点

| 问题 | 现象 | 解决 |
|------|------|------|
| 预设编辑器忘记初始化 | 列表为空 | 先点「清空并初始化列表」 |
| L→R 镜像不完整 | 右手骨骼缺失 | 镜像基于从左到右的文本替换逻辑（如 `_L` → `_R`），如果命名不符合此规律则需手动填写 |
| 保存前忘记输入名称 | 找不到保存文件 | 在「保存名」输入框中输入文件名后再点保存 |
| 假头法缺少原生骨架 | 下拉菜单显示「无可用骨架」 | 将对应 fbxskel 文件放入 `assets/native_skeletons/re4/` 目录 |
| 面部权重简化选错对象 | 功能无法使用 | 需要选中一个网格对象（MESH），不是骨架 |
| RE9 同步同时选中父子 | 操作失败并报错 | 只选中每个链的链首骨骼，不要同时选中子骨 |

### 涉及的面板区域

- 预设编辑器标签页（全部）
- 姿态转换区（录制/正向/逆向）
- RE4 Tools（假头法）
- MHWilds Tools（Endfield 面部改名、面部权重简化）

- 基础工具区（镜像对齐 X、添加尾骨）
